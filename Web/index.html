O que é a Web

Beleza. Mas por onde começar?

Nada como começar pelo começo, então vamos falar de como a web virou essa coisa maravilhosa que utilizamos. Primeiro precisamos saber o que é internet e o que é Web.

A internet nasceu durante a guerra fria, devido a necessidade norte americana de manter a sua hegemonia, então foi investido em melhorar a comunicação criando uma rede cabeada que ligava 4 computadores distintos e que estavam em lugares diferentes do país, um deles militar, essa rede se chamava ARPA (Advanced Research Projects Agency). Para fazer a comunicação foi criado um padrão de comunicação que não dependia do modelo das máquinas presentes na rede, assim nasceu o primeiro protocolo o NCP (Network Control Protocol) e posteriormente outros protocolos começaram a surgir com objetivos diversos, dentro desse conjunto de protocolos dois deles ainda são utilizados até hoje, que são o FTP (e-mail) e o DNS (atribuição de nomes). Com o passar do tempo essa rede foi aumentando até envolver todo o planeta. tipo assim:

Quatro pessoas muito importante para a criação da internet como a conhecemos hoje são, Ray Tomlinson o criador da idéia de e-mail e quem definiu a utilização do @ para separar nome do usuário do domínio (usuário@domínio), Robert Kahn idealizador do protocolo TCP (Transfer Control Protocol) que estabelece a passagem de dados através de pacotes, Vint Cerf o idealizador do IP (Internet Protocol) que define um endereço para cada dispositivo na rede e por o Tim Bernes Lee, criador da linguagem de marcação HTML e do protocolo HTTP (HyperText Transfer Protocol). A Internet é uma rede mundial de conexões de computadores (Rede de redes), através dessas conexões existem comunicações utilizando diferentes protocolos e a WEB se caracteriza por seguir esses 3 protocolos específicos para a comunicação (TCP, IP, HTTP/HTTPS). Mas da onde vem essas informações ? Para onde vão? Quem recebe ? Quem envia? Sexta, no globo repórter! As principais partes que se comunicam e fazem a magia acontecer através de tantos protocolos são os Servidores e os Clientes.

Cliente x Servidor


Servidores

Os servidores são os “gigantes”, mas em tamanho mesmo pois eles precisam de grande capacidade de processamento de informações com o objetivo de receber e responder informações requisitadas pelos clientes, clientes esses que são respondidos simultaneamente e de qualquer lugar do mundo (serviços distribuídos). Quando ao software desses equipamentos, ele é preparado para estar sempre em execução esperando uma requisição do cliente. Mas não se engane servidores não necessariamente são máquinas enormes dentro de corporações, eles também podem ser locais, como um servidor privado para uma empresa ou até um servidor local na sua máquina.

Clientes

Os clientes ficam fazendo requisições o tempo todo para os servidores, eles que são responsáveis pelo começo e pelo fim dessa comunicação (existem casos que o servidor pode corta a comunicação por motivos de segurança), no contexto da web um cliente não pode se comunicar com outro cliente como em outros protocolos, e as máquinas do usuário (que são os clientes) tornam a internet, rede e todos os protocolos invisíveis ao usuário.

E para se comunicar bem o HTTP solta o verbo!

Os verbos são utilizados para fazer requisições para os servidores web, os principais verbos que utilizados são: GET (para chamar uma página, links, imagens e etc), POST (utilizado para fazer uma modificação, como envio de um formulário ou atualizações de informações), DELETE (para fazer a exclusão de algo), PUT (substitui todas as atuais representações), entre outros métodos que podem ser vistos aqui https://www.w3.org/Protocols/rfc2616/rfc2616-sec5.html#sec5, outro documento interessante porém mais técnico que e de boa leitura e a Documentacao RFC 2616: https://tools.ietf.org/html/rfc2616 .

Já está quase acabando! Prometo, mas precisamos esclarecer o que é Resquests e Reponse

Primeiro Request

Request é a parte em que o seu lado trabalha, o seu browser faz uma requisição via GET além de outras informações que são mandadas como por exemplo:O corpo de HTTP, o número de Bytes enviados pelo cliente, coleção de cabeçalhos, a porta que ele está utilizando, qual porta que irá entrar no servidor, o estado e nome do servidor e faz uma verificação de conexão de SSL (HTTP + SSL = HTTPS!!!).

Response como o nome já surgere nele cotem a resposta para todas as informações pedidas pelo cliente.

Entender dos protocolos e como eles funcionam pode ser o caminho para abrir muitas Portas!

Portas como as portas utilizadas por nós clientes,temos uma porta padrão, geralmente para requisições de formato HTTP as portas utilizadas são a 80 (uma porta conhecida, de baixa numeração) e a 8080, já para requisições HTTPS por exemplo a porta padrão utilizada é a 443, ambas requisições utilizam os protocolos TCP/UDP para caracterizar a forma de passar arquivos e IP para endereçar saída e destino.

Hummmm mas o que é esse tal de HTTP e HTTPS ???

O protocolo HTTP tem como principal função fazer transferência de mídias como, textos, imagens, links e etc, por isso tem o nome de Hypertext Transfer Protocol (um conceito criado antes mesmo do surgimento da internet), um derivado dessa protocolo é o HTTPS que nada mais é do que uma forma mais segura de fazer essa transferência pois antes de fazer a requisição ele criptografa as informações de serem enviadas. Medida tomada para suprir uma falha na arquitetura da rede em si, para saber mais procure o livro: Redes de Computadores do Kurose.

Mas e aqueles protocolos lá? TCP/UDP?

O protocolo TCP(Transmission Control Protcol) é responsável também por estabelecer e garantir a conexão com o servidor através do método (3 way handshake), nesse método somente o cliente pode começar a conversa, porém ambos os lados podem decidir terminar a conexão (Pesquise!).

Já o protocolo UDP(User Datagram Protocol) é um protocolo não orientado para a conexão da camada de transporte do modelo TCP/IP. Este protocolo é muito simples já que não fornece controle de erros (não está orientado para a conexão).

Está com fome? Que bom! por que vamos falar rapdinho de Cookies!

O cookie método que os browsers utilizam para fazer um melhor reconhecimento de quem está utilizando aquela rede e de como recuperar uma tarefa que estava sendo executada, eles são gerados pelo servidor quando o cliente faz o primeiro contato com o servidor (Aff, achei que era outra coisa...), nesse contato o servidor coloca uma identificação no cliente para saber se o cliente já esteve ali, e quais eram as tarefas que estavam sendo desempenhadas por ele. Veja como funciona um cookie na linguagem JavaScript: https://www.w3schools.com/js/js_cookies.asp.

URL/URN/URI três siglas que ninguém sabe ao certo qual a diferença

Quando aquele endereço que digitamos na barra do navegador para irmos parar em algum site que comumente chamamos de URL (Uniform Resource Locator), não está 100% correto utilizar esse termo, por a URL tratar-se apenas do domínio de fato daquele site, por exemplo: infojr.com.br é a URL da infojr, porém também se tem a URN (Uniform Resource Name) que identifica o recurso que será utilizado, exemplo: URL + URN = infojr.com.br/home.html ou infojr.com.br/contatos.php. E por último mas não menos importante temos a URI (Uniform Resource Identifier) que como o próprio nome diz identifica o recurso, sendo ele uma imagem, um formulário e etc, e também traz o protocolo que está sendo usado, exemplos de URI são:https://www.w3.org/Protocols/HTTP/1.1/rfc2616.pdf e https://infojrufba.github.io/estudos/.

Nessa imagem podemos ver algumas coisas que já comentamos, mas falta esse Status code mas é algo bem simples, se liga!

O status code tem várias famílias, mas podemos resumi-las à uma lista:

  • família 1xx é uma resposta de informações como por exemplo a reposta 100(continue), ou 102(Trocando de protocolos).
  • família 2xx resposta de sucesso! Por exemplo a resposta 200(OK), ou 202(Aceito/Accepted).
  • família 3xx resposta de Redirecionamento. Por exemplo 300(multiplas escolhas), ou 302(encontrado).
  • família 4xx reposta de Erro no lado do cliente. Por exemplo 404(Não econtrado), ou 408(Request timeout).
  • família 5xx resposta de Erro no lado do Servidor. Por exemplo 500(Um erro interno de servidor), ou 502(bad Gateway)

E para finalizar! Web server ou Web services?

Basicamente um web server é um servidor que retorna páginas HTML, PHP, contendo diferentes formatos de arquivo como, png, jpeg, pdf e etc, para um cliente na web. Já um webservice é o serviço em si, o site, o aplicativo, o sistema que faz uso da web.
